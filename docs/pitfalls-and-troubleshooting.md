# Pitfalls and Troubleshooting

A record of all issues encountered during development, their root causes, and solutions. This document is especially useful for future debugging.

## Critical Pitfalls

### 1. CGEventTap on background thread doesn't receive Bluetooth events

**Symptom:** USB keyboard media keys work, but AirPods don't generate any events.

**Cause:** A CGEventTap added to a background thread's run loop doesn't receive `NX_SYSDEFINED` events originating from Bluetooth devices on modern macOS.

**Solution:**

```cpp
// CORRECT: main run loop
CFRunLoopAddSource(CFRunLoopGetMain(), eventSource, kCFRunLoopCommonModes);

// WRONG: background thread run loop
CFRunLoopAddSource(CFRunLoopGetCurrent(), eventSource, kCFRunLoopCommonModes);
```

---

### 2. AVAudioApplication handler registers but never fires

**Symptom:** `setInputMuteStateChangeHandler` returns success (BOOL YES, no error), but the callback is never executed when the user presses the AirPods stem.

**Cause:** The handler is only invoked when the app has **active audio I/O**. Registering the handler alone is not sufficient.

**Solution:** Create an AUHAL audio unit that pulls data from the microphone (and discards it). See `setupAUHAL()` in `media_key_tap.cc`.

**Evidence:** Apple's `AVAudioApplication.h` header:

> "this notification will only be dispatched for state changes when there is an active record session"

---

### 3. macOS disables event tap by timeout

**Symptom:** Media keys stop working after some time without use.

**Cause:** macOS automatically disables event taps that get "stuck" or don't respond quickly enough.

**Solution (two layers):**

1. In the callback, handle `kCGEventTapDisabledByTimeout`:

```cpp
if (type == kCGEventTapDisabledByTimeout) {
    CGEventTapEnable(eventPort, true);
    return event;
}
```

2. Health check timer every 5 seconds:

```cpp
if (eventPort && !CGEventTapIsEnabled(eventPort)) {
    CGEventTapEnable(eventPort, true);
}
```

---

### 4. "Cannot Control Mic" notification cannot be filtered

**Symptom:** The notification appears even with the handler registered.

**Cause:** The notification is generated by `audioaccessoryd` and is not a `UNNotification` that can be intercepted or hidden. The only way to suppress it is to have the `AVAudioApplication` handler active and returning `YES`, with active audio I/O.

**Solution:** AUHAL + AVAudioApplication handler returning YES (the full pipeline).

---

### 5. Double-toggle of mute (mutes and unmutes instantly)

**Symptom:** When pressing the AirPods stem, Meet mutes and immediately unmutes (or vice versa).

**Cause:** Multiple event sources fire for the same gesture:

- CGEventTap + NSEvent for media keys
- AVAudioApplication + Darwin notification for AirPods mute

**Solution:** Deduplication based on atomic timestamps:

- 200ms window for CGEventTap + NSEvent
- 500ms bidirectional window for AVAudioApplication + Darwin

---

### 6. AirPods press-and-hold doesn't generate NX_SYSDEFINED

**Symptom:** Single tap works, but press-and-hold doesn't generate any events in CGEventTap.

**Cause:** Press-and-hold is a proprietary mute gesture that follows a completely different path in macOS (Bluetooth proprietary → audioaccessoryd). It's not a media key.

**Solution:** Use Darwin notification and/or AVAudioApplication handler, which are the only paths for detecting this gesture.

---

### 7. NSEvent monitor cannot consume events

**Symptom:** Events intercepted by the NSEvent fallback pass through to other apps (Spotify pauses/resumes).

**Cause:** `addGlobalMonitorForEventsMatchingMask:` is read-only — it cannot modify or consume events. Only CGEventTap can.

**Impact:** If CGEventTap captures the event, it's consumed. If only NSEvent captures it, the event reaches other apps. In practice, this is rarely a problem because CGEventTap is the primary source.

---

## Troubleshooting by Log

MeetPods uses tags in all logs for easy filtering:

| Tag                    | Component                          |
| ---------------------- | ---------------------------------- |
| `[MeetPods:native]`    | C++/ObjC++ addon                   |
| `[MeetPods:media-key]` | TypeScript addon wrapper           |
| `[MeetPods:bridge]`    | WebSocket server                   |
| `[MeetPods:main]`      | Main orchestrator (index.ts)       |
| `[MeetPods:bg]`        | Background service worker          |
| `[MeetPods:content]`   | Content script in Meet             |

### Expected logs at startup

```
[MeetPods:main] Accessibility permission: GRANTED
[MeetPods:main] Microphone permission: granted
[MeetPods:native] Event tap created successfully
```


## Diagnostic Checklist

If something isn't working, follow this order:

1. **Accessibility permission?**
   - Log: `Accessibility permission: GRANTED` or `NOT GRANTED`

2. **Event tap created?**
   - Log: `Event tap created successfully` or `FAILED to create event tap`

3. **Extension connected?**
   - Log: `Client connected (total: N)`

4. **Meet detected?**
   - Log: `meet-status push: active=true`

5. **AUHAL active?** (for AirPods mute gesture)
   - Log: `AUHAL: audio input started`

6. **Microphone permission?**
   - Log: `Microphone permission: granted` or `denied`

7. **Events arriving?**
   - Media key: `Play/Pause key event (CGEventTap)`
   - AirPods mute: `AirPods mic mute event (AVAudioApplication)` or `Darwin notification: AirPods mute gesture detected`

8. **Toggle executed?**
   - Log: `handleMediaKey() — toggling mute...`
   - Log: `toggle result: success=true, muted=<state>`
